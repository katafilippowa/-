# Проектирование витрины данных: marketing_channel_effectiveness
## Группа Б | Анализ эффективности маркетинговых кампаний

---

## ШАГ 1. Бизнес-задача

**Вопрос:** Какие каналы привлечения приносят наиболее качественных клиентов?

**Критерии качества клиента:**

| Критерий | Метрика | Почему важно |
|----------|---------|--------------|
| Высокий LTV | Сумма всех платежей клиента | Прямая финансовая ценность для бизнеса |
| Низкий отток | Длительность жизни клиента (месяцев) | Удержание = стабильный доход, ниже CAC |
| Низкая нагрузка на поддержку | Среднее количество тикетов на клиента | Меньше операционных затрат на обслуживание |

**Целевая витрина:** `marketing_channel_effectiveness.csv`

> ⚠️ **ВАЖНО для Практической работы №2:** Витрина должна содержать данные для:
> - Ранжирования каналов по комплексному индексу качества
> - Построения когортного анализа (по месяцам привлечения)
> - Расчета прогнозной ценности клиента (LTV)
> - A/B-тестирования гипотез по оптимизации каналов
> - Визуализации в дашбордах (графиков, таблиц, фильтров)

## ШАГ 2. Уровень агрегации

**Выбранный уровень:** `Канал + Месяц привлечения (acquisition_month)`

**Обоснование выбора:**

| Вариант | Плюсы | Минусы | Почему не подходит |
|---------|-------|--------|-------------------|
| Транзакция | Максимальная детализация | Миллионы строк, сложно агрегировать | Не отвечает на вопрос "какой канал лучше" |
| Компания | Просто, один клиент = одна строка | Теряется временной срез для когорт | Нельзя отследить динамику качества во времени |
| **Канал + Месяц** ✅ | Баланс детализации и аналитичности | Требует предварительной агрегации | **Оптимально для сравнения каналов во времени** |
| Канал (итого) | Максимально просто | Нельзя увидеть тренды и сезонность | Недостаточно для глубокого анализа |

**Одна строка витрины =** Агрегированные метрики по одному маркетинговому каналу за один месяц привлечения лидов.

## ШАГ 4. Таблица структуры витрины

| Поле | Тип | Источник | Описание | Способ расчета (если расчетное) |
|------|-----|----------|----------|--------------------------------|
| channel | string | 07_marketing_leads | Нормализованное название канала | Приведение к lowercase, удаление спецсимволов |

| period_end | date | Расчетное | Конец периода | LAST_DAY(acquisition_month) |

## ШАГ 5. Техническое задание для Data Engineering

### 5.1. Цель создания витрины
Создать агрегированную витрину данных `marketing_channel_effectiveness` для анализа эффективности маркетинговых каналов привлечения клиентов. Витрина должна позволять сравнивать каналы по критериям: доходность (LTV, revenue), качество клиентов (удержание, активность), нагрузка на поддержку и рентабельность (ROI, CAC). Данные должны быть готовы для использования в когортном анализе и построении дашбордов в Практической работе №2.

### 5.2. Источники данных
- **Основной источник:** `07_marketing_leads.csv` (каналы, кампании, лиды, затраты)
- **Финансовые данные:** `05_finance_payments.csv` (платежи, выручка, статусы)
- **Данные поддержки:** `06_support_tickets.csv` (тикеты, оценки удовлетворенности, время ответа)
- **Справочник сотрудников:** `08_hr_employees.csv` (косвенно, для определения ответственных)
- **Период данных:** 2019-01-01 по текущую дату
- **Ключ связи:** `company_id` (приведенный к единому формату во всех источниках)

### 5.3. Требования к детализации и периоду
- **Гранулярность:** Уровень `канал + месяц привлечения` (acquisition_month)
- **Временной период:** Помесячная агрегация с возможностью кумулятивного расчета (нарастающим итогом)
- **Исторические данные:** Полная история с 2019 года
- **Обновление:** Инкрементальное добавление данных за прошедший месяц
- **Задержка данных (latency):** Не более 24 часов с момента появления данных в источниках

### 5.4. Бизнес-правила
1. **Нормализация каналов:** Привести все названия каналов к единому справочнику:
   - "Google Ads", "google_adwords", "Google" → "google_ads"
   - "Facebook", "fb", "facebook.com" → "facebook"
   - "Email", "email_marketing" → "email"
   
2. **Фильтрация платежей:** Учитывать только успешные платежи (amount > 0, status != 'refunded', status != 'cancelled')

3. **Расчет LTV:** Сумма всех платежей клиента за все время наблюдения (без дисконтирования)

4. **Расчет конверсии:** Конвертированным считается лид, у которого есть хотя бы один платеж в течение 90 дней после даты лида

5. **Обработка пропусков:**
   - `campaign_cost`: NULL → 0
   - `satisfaction_score`: исключить из расчета среднего, если значение отсутствует
   - `response_time`: рассчитывать только для закрытых тикетов

6. **Расчет quality_score:**
7. quality_score = (avg_ltv / NULLIF(cac, 0)) × (1 / NULLIF(avg_tickets_per_client, 0))
8. Значение нормализовать к диапазону 0-100 для удобства визуализации

7. **Дедупликация:** При наличии дубликатов `lead_id` оставлять запись с максимальной `lead_date`

- **Расположение:** `/data/marts/marketing/`
